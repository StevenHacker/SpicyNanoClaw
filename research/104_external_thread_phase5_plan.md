# External Thread Phase 5 Plan

## Purpose

Phase 5 is the first external-thread wave designed specifically for `Milestone 2`.

It does not reopen broad repo mapping.
It supports the next engineering cuts:

- controller-issued worker launch
- host seam confidence
- donor lifecycle precision
- clean delivery/product envelope

## Packet Set

Carry-forward precision packets:

1. `25` OC-10 Runner Lifecycle Timing Matrix
2. `26` OC-11 Plugin SDK / Slot Stability Atlas
3. `27` CC-10 Pressure / Compaction Lifecycle Matrix
4. `28` CC-11 Memory Lifecycle Contract Matrix

New Milestone-2-directed packets:

5. `29` OC-12 Worker Invocation Seam Matrix
6. `30` OC-13 Plugin Delivery / Marketplace Rehearsal Packet
7. `31` CC-12 Delegation Ownership / Addressed Queue Matrix
8. `32` SYN-06 SNC Milestone 2 Product Envelope

## Why These Packets

### `25`

Still needed because worker launch, durable memory, shaping, and maintenance all get more dangerous when runner timing is assumed instead of proven.

### `26`

Still needed because `Milestone 2` is where SNC starts leaning harder on long-lived extension seams rather than only one bounded plugin cut.

### `27`

Still needed because later pressure-control imports should come from exact lifecycle knowledge, not from principle summaries alone.

### `28`

Still needed because durable memory is now live and future controls should align with the real donor contract.

### `29`

This is the most direct OpenClaw packet for `Milestone 2`.
It should close the remaining ambiguity between:

- host worker tools
- hook timing
- completion delivery
- practical controller-launch seams

### `30`

This packet supports product reality rather than runtime theory.
It should tell us what a clean install/update/publication path actually looks like for an OpenClaw plugin like SNC.

### `31`

This packet sharpens the CC donor side for multi-worker behavior.
The project already knows the high-level donor principles; now it needs the exact ownership/queue/addressing mechanics that are still worth learning from.

### `32`

This packet synthesizes the `Milestone 2` product boundary so development does not accidentally widen into a mini-platform rewrite.

## Best Bundles

- `25 + 29`
  - best OpenClaw kernel/support bundle for worker-launch engineering

- `27 + 31`
  - best CC donor bundle for orchestration and pressure-control precision

- `30 + 32`
  - best product/delivery bundle once runtime behavior is clearer

Best kept solo:

- `26`
  - slot/sdk stability benefits from one coherent read

- `28`
  - memory lifecycle stays cleaner when not mixed with broad donor interpretation

## Expected Progress Lift

These are planning estimates, not victory theater.

### OpenClaw

Current read:

- SNC-relevant host understanding: about `90%`
- broader host/platform understanding: about `80%`

If `25 + 26 + 29 + 30` close well:

- SNC-relevant host understanding should move to about `95%`
- broader host/platform understanding should move to about `86%`

Why not higher:

- after this point, many remaining gains come from direct implementation contact rather than more reading

### CC

Current read:

- SNC-relevant donor understanding: about `88%`
- broader repo/product understanding: about `77%`

If `27 + 28 + 31` close well:

- SNC-relevant donor understanding should move to about `93%`
- broader repo/product understanding should move to about `83%`

Why not higher:

- the remaining repo bulk is increasingly peripheral to SNC and the specialization-kernel program

## Success Condition

Phase 5 is successful when:

1. the worker-launch path stops depending on fuzzy host assumptions
2. the remaining donor imports for pressure/memory/delegation are lifecycle-accurate
3. delivery/product-envelope planning is concrete enough to support `Milestone 2` without guesswork
4. external research keeps tightening implementation decisions instead of widening scope
